실행:

```sh
cargo run --example 14_handmade_map_geometry
```

![직접 만든 맵 지오메트리 예제는 보이는 바닥 타일, 충돌하는 벽 사각형, 충돌 경계 안의 플레이어를 보여줍니다.](../../assets/screenshots/ch14-handmade-map.png)

이 장의 계약은 보이는 바닥과 충돌하는 벽을 분리하는 것입니다. 바닥 타일은 시각 요소만 갖고, 벽은 `Wall + Body + Transform`을 가져 충돌 시스템의 대상이 됩니다.

## 핵심 ECS 계약

- `Wall`: 충돌 가능한 정적 지형입니다.
- `Body { half_size }`: 플레이어와 벽이 공유하는 AABB 크기입니다.
- `Velocity(Vec2)`: 입력 시스템이 쓰고 이동 시스템이 읽습니다.
- `PlayerBundle`: 플레이어, 바디, 속도, 스프라이트를 묶습니다.
- `WallBundle`: 벽 마커, 바디, 색상 스프라이트를 묶습니다.

시스템 세트는 `Input -> Movement -> Collision`입니다. 먼저 의도를 만들고, 위치를 바꾼 뒤, 겹친 벽에서 플레이어를 밀어냅니다.

## Rust 포인트

`for (position, size) in [...]` 배열은 맵 데이터를 코드 안에 직접 둡니다. 이 단계에서는 명시적 데이터가 벽의 위치와 크기를 가장 읽기 쉽게 보여줍니다.

`delta.abs()`는 `Vec2`의 각 축 절댓값을 구합니다. `overlap = player_half + wall_half - delta.abs()`가 양수이면 해당 축에서 겹친 것입니다.

## Bevy 포인트

이 예제의 충돌은 축 정렬 박스 충돌입니다. 겹친 축 중 더 작은 축으로 플레이어를 밀어내면 벽을 따라 미끄러지는 기본 동작을 만들 수 있습니다.

바닥은 `Sprite::from_color`만 있고 `Body`가 없습니다. ECS에서는 어떤 시스템이 어떤 컴포넌트를 조회하는지가 곧 기능 참여 조건입니다. 충돌에 참여하려면 `Body`와 `Wall`이 있어야 합니다.

## 프레임 흐름

1. 입력이 플레이어 `Velocity`를 씁니다.
2. 이동 시스템이 위치를 갱신합니다.
3. 충돌 시스템이 플레이어와 모든 벽을 비교합니다.
4. 겹침이 있으면 더 작은 축 방향으로 플레이어를 밀어냅니다.
5. 바닥 타일은 렌더링을 담당하고, 벽 엔티티가 충돌 계산을 담당합니다.

## 흔한 실수

- 벽에는 `Body`와 `Wall`을 함께 붙입니다. `With<Wall>` 쿼리는 `Wall` 마커가 있는 엔티티를 선택합니다.
- 충돌을 이동보다 먼저 실행하면 이번 프레임 이동으로 생긴 겹침을 해결하지 못합니다.
- `signum()`은 `delta`가 0이면 0을 돌려줍니다. 완전히 같은 중심에 겹치는 특수 상황은 별도 처리가 필요할 수 있습니다.
- 시각 크기와 `Body.half_size`가 다르면 보이는 벽과 실제 충돌 위치가 어긋납니다.

## 작게 바꿔보기

- 벽 사각형을 하나 더 추가하세요.
- 플레이어 크기를 키우고 충돌 반응을 확인하세요.
- 벽이 아닌 exit zone trigger를 추가해보세요.
